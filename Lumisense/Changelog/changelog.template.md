# Шаблон changelog.json

Этот файл — справочник по формату `changelog.json`, лежащего рядом с ним в этой же
папке. Ничего запускать не нужно, это просто шпаргалка по полям: что писать и как
это будет выглядеть в окне «Список изменений» в программе.

Нужен просто пустой шаблон для копирования, без пояснений — он рядом,
`Changelog/changelog.template.json`.

`changelog.json` копируется рядом с `.exe` при каждой сборке (см. `Lumisense.csproj`
и комментарий в `ChangelogLoader.cs`). Редактировать можно прямо в папке со сборкой —
пересобирать проект не обязательно, изменения подхватятся при следующем запуске
программы.

## Главное: номер версии писать не нужно — он считается по смыслу изменений

Номер версии (`"1.0.0"`, `"1.1.0"`, `"2.0.0"`, ...) программа считает **сама** — поля
`version` в файле нет. Считается он в стиле semver (`major.minor.patch`) и зависит не
просто от того, что запись есть, а от того, **что именно лежит в её списке
изменений**:

1. Все записи сортируются по полю `date` от самой старой к самой новой.
2. Самая старая — это база отсчёта, версия `1.0.0`.
3. Дальше для каждой следующей записи по порядку смотрим на типы (`type`) её строк
   изменений и увеличиваем номер по правилу SemVer (сверху вниз, срабатывает первое
   подходящее):
   - есть хоть одна строка с `"type": "removed"` (что-то убрали, ломает обратную
     совместимость) → **major** (`X.0.0` — минор и патч сбрасываются в 0);
   - иначе есть хоть одна с `"type": "added"` или `"type": "changed"` (новая или
     изменённая функциональность, но без потери обратной совместимости) → **minor**
     (`X.Y.0` — патч сбрасывается в 0);
   - иначе (только `"fixed"` — по SemVer патч зарезервирован именно за багфиксами,
     либо список изменений вообще пустой) → **patch** (`X.Y.Z`, увеличивается сам).
4. **«Текущая версия»** — это запись с самой свежей датой, а не первая или последняя
   по расположению в файле. Порядок записей в самом JSON значения не имеет — можно
   дописывать новые версии как в конец файла, так и в начало, программа разберётся
   по датам сама.

### Пример

```json
[
  { "date": "Первый релиз", "changes": [
      { "type": "added", "text": "Базовый плеер" }
  ]},
  { "date": "2026-06-01", "changes": [
      { "type": "fixed", "text": "Починили падение при пустом плейлисте" }
  ]},
  { "date": "2026-06-20", "image": "mini-player.png", "changes": [
      { "type": "added", "text": "Добавили мини-плеер" }
  ]},
  { "date": "12 июля 2026", "changes": [
      { "type": "removed", "text": "Убрали старый формат плейлиста, теперь он несовместим" }
  ]}
]
```

После автоматического подсчёта (по датам от старой к новой) получится:
`1.0.0` → `1.0.1` (был только `fixed`) → `1.1.0` (был `added`) → `2.0.0` (был
`removed`, самая свежая дата — 12 июля 2026, она же «Текущая версия»). Этот же номер
автоматически попадает и в карточку «О программе» в настройках («Версия {version}») —
отдельно его нигде вписывать не нужно, он всегда берётся из записи, помеченной как
текущая (см. `RefreshAppVersionText` в `SettingsWindow.xaml.cs`).

## Поля одной версии

| Поле      | Тип             | Обязательно | Где отображается |
|-----------|-----------------|-------------|-------------------|
| `date`    | строка          | да          | Подзаголовок под заголовком версии (мелким блёклым текстом); также используется для сортировки записей и определения текущей версии |
| `image`   | строка          | нет         | Картинка версии в панели деталей справа, над списком изменений |
| `changes` | массив объектов | да          | Сам список изменений этой версии, сгруппированный по `type`; также определяет, какую часть номера версии увеличить (см. выше) |

**`date`** — можно писать что угодно, например `"Первый релиз"` (так и показывается
в текущем `changelog.json`). Но программа попробует распознать в этой строке
настоящую дату — понимает ISO (`"2026-07-12"`) и русский формат с названием месяца
(`"12 июля 2026"`). От результата этого распознавания зависят:

- в каком порядке записи проходятся при подсчёте версий (от старой к новой);
- какая запись получит бейдж «Текущая версия» (та, чья дата самая свежая);
- порядок при сортировке списка версий по кнопке «Дата» в окне списка изменений (второй
  вариант сортировки в этом окне — «Кол-во», по числу пунктов изменений в версии; отдельной
  сортировки «по версии» нет, потому что номер версии как раз и вычисляется по порядку дат —
  такая сортировка была бы точной копией сортировки по дате).

Если распознать дату не получилось (как с «Первый релиз») — запись считается самой
старой из всех. Если такой нераспознанной записью окажется единственная в файле, ей
всё равно ставится бейдж «Текущая версия» — конкурентов на "самую свежую дату" нет.

**`image`** — необязательное поле, можно вписать двумя способами:

- прямая ссылка на картинку в интернете — `"https://example.com/screenshot.png"`;
- имя файла (или относительный путь), который лежит рядом с `changelog.json`, то есть
  прямо в этой же папке `Changelog` — например `"release-1.2.png"` или
  `"screenshots\dark-theme.png"` (сам файл картинки нужно положить в `Changelog`
  рядом, программа его туда не копирует и не скачивает).

Если поле не заполнено или картинка не найдена/не загрузилась — блок с картинкой
просто не показывается, на остальную часть версии это не влияет.

## Поля одной строки изменения (`changes[]`)

| Поле    | Тип    | Обязательно | Где отображается |
|---------|--------|-------------|-------------------|
| `type`  | строка | нет         | Определяет иконку, подпись секции, цвет карточки — и то, какую часть номера версии увеличит эта запись (см. выше) |
| `text`  | строка | да          | Сам текст строки изменения, внутри своей карточки |
| `image` | строка | нет         | Картинка именно этого пункта — показывается внутри его карточки, под текстом (не путать с полем `image` версии — оно про картинку сразу для всей версии, целиком, см. выше) |

**`type`** — один из четырёх ключей ниже (регистр не важен). Каждому соответствуют
своя иконка, подпись секции на русском и акцентный цвет:

| Ключ (`type`) | Подпись в программе | Иконка         | Цвет                        | Влияние на версию      |
|---------------|----------------------|----------------|------------------------------|--------------------------|
| `added`       | Добавлено            | `IconAdd`      | зелёный `#22C55E`            | minor (если нет removed) |
| `changed`     | Изменено             | `IconEdit`     | синий `#3B82F6`               | minor (если нет removed) |
| `fixed`       | Исправлено           | `IconWrench`   | оранжевый `#F59E0B`          | сам по себе — patch      |
| `removed`     | Удалено              | `IconDelete`   | красный `#EF4444`            | major (перекрывает всё)  |

Если `type` не указан, пустой или это что-то незнакомое (опечатка, старый формат) —
программа не ломается, а молча показывает строку как «Изменено» (нейтральный синий,
влияет на версию как minor, см. таблицу выше).

Строки изменений внутри одной версии автоматически группируются по `type` — то есть
в JSON их порядок и чередование значения не имеют, в программе они всё равно соберутся
в отдельные блоки «Добавлено» / «Изменено» / «Исправлено» / «Удалено» (в этом порядке,
показываются только те блоки, для которых реально есть строки). На номер версии влияет
только сам факт присутствия того или иного типа в записи — сколько раз он встретился,
значения не имеет: одно `"removed"` даёт такой же скачок major, как и десять.

**`text`** — сам текст пункта, показывается как есть, переносится на несколько строк
при необходимости. Длина не ограничена, но для читаемости лучше держать одну мысль
в одном пункте, как в примерах выше.

**`image`** (у пункта, не у версии) — необязательная картинка именно этого конкретного
изменения, например скриншот новой кнопки, о которой идёт речь в `text`. Заполняется
точно так же, как и `image` версии — ссылкой или именем файла рядом с `changelog.json`
в папке `Changelog`. Показывается внутри карточки этого пункта, под текстом; если не
задана — карточка выглядит как раньше, просто текст без картинки.

## Если файл отсутствует или повреждён

Если `changelog.json` не найден рядом с `.exe` или не читается (битый JSON и т.п.),
программа не падает и не показывает пустое окно — вместо этого используется
встроенный запасной список из одной версии (он зашит в `ChangelogLoader.cs`, в методе
`DefaultEntries()`), и её номер версии и бейдж «Текущая версия» точно так же
считаются автоматически, по тем же правилам.
